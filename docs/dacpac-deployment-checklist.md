# DACPAC Deployment Checklist

A practical checklist for SQL Server DACPAC deployments. It is intentionally focused on boring, repeatable deployment work — because boring deployments are the best deployments.

## Before deployment

### Environment

- Verify target SQL Server version and edition.
- Verify database compatibility level.
- Confirm whether the target is SQL Server, Azure SQL Database or SQL Managed Instance.
- Check free disk space for data, log and backup locations.
- Confirm latest successful backup exists and can be restored.
- Confirm maintenance window and rollback plan.

### Database state

- Check database size and transaction log size.
- Check disabled indexes.
- Check disabled or untrusted foreign keys.
- Check active blocking and long-running transactions.
- Check orphaned users and login dependencies.
- Review permissions, users, roles and logins generated by the deployment.

### DACPAC review

- Review generated deployment script before running it.
- Watch for destructive changes such as DROP TABLE, DROP COLUMN or data type narrowing.
- Check whether large tables are rebuilt or copied.
- Check index drops and recreations.
- Check foreign key drops and recreations.
- Verify pre-deployment and post-deployment scripts are idempotent.
- Verify environment-specific values are not hardcoded.

## During deployment

- Capture full deployment output.
- Monitor blocking sessions.
- Monitor transaction log growth.
- Monitor disk space.
- Keep the deployment script and publish profile used for the deployment.

## After deployment

- Verify schema version or application version marker.
- Run smoke-test queries.
- Check disabled indexes and foreign keys again.
- Check untrusted foreign keys again.
- Review deployment warnings.
- Verify application startup and basic workflows.
- Keep deployment logs with the release notes.

## Useful SqlPackage options

These options are often useful, depending on the environment and deployment policy:

```powershell
SqlPackage.exe /Action:Publish `
  /SourceFile:"path-to-database.dacpac" `
  /TargetServerName:"server\\instance" `
  /TargetDatabaseName:"database" `
  /TargetTrustServerCertificate:True `
  /p:DropObjectsNotInSource=False `
  /p:IgnorePermissions=True `
  /p:IgnoreRoleMembership=True `
  /p:IgnoreLoginSids=True `
  /p:ExcludeObjectType=Logins `
  /p:ExcludeObjectType=Users
```

## Notes

- Do not blindly reuse options across environments.
- `DropObjectsNotInSource=False` is usually safer for shared or manually extended databases.
- Ignoring permissions/users/logins can help avoid environment-specific deployment failures, but security must still be managed intentionally.
- Always review the generated script for high-risk operations before applying it to production.
